## 1. Назва модуля

**Visual Quiz Delivery**

Модуль є платним розширенням для **API Quiz Bank**.

Його задача — доставляти вікторини не тільки у текстовому форматі, а й у форматі:

```text
зображення + питання + варіанти відповіді у Telegram quiz poll
```

---

## 2. Коротке визначення

**Visual Quiz Delivery** — це модуль, який бере готову вікторину з банку, створює або знаходить для неї відповідне зображення, перевіряє якість і доставляє результат у Telegram-канал або бот.

Модуль не замінює Quiz Bank.  
Він підсилює його та створює дорожчий формат підписки.

---

## 3. Продуктова ціль

Головна ціль:

**зробити вікторини візуально сильнішими, дорожчими для клієнта і придатними для продажу як premium-функція.**

Модуль має допомогти:

- підняти цінність API Quiz Bank;
- створити додатковий тариф;
- збільшити залучення аудиторії в Telegram;
- дати клієнтам вибір між простим і візуальним форматом;
- повторно використовувати вже створені візуальні матеріали через кеш у межах прав, стилю та consumer boundaries.

---

## 4. Для кого цей модуль

### Основні клієнти

1. Власники Telegram-каналів з мовним контентом.
2. Власники освітніх ботів.
3. Автори мікрокурсів.
4. Адміністратори навчальних спільнот.
5. Майбутні B2B-клієнти, яким потрібен готовий навчальний контент.

---

## 5. Яку проблему вирішує

Звичайна текстова вікторина часто виглядає сухо.

Клієнту потрібно:

- швидко отримувати готовий контент;
- не наймати дизайнера;
- не створювати картинки вручну;
- мати стабільний стиль;
- мати більше уваги від аудиторії;
- публікувати красивіші матеріали без зайвої роботи.

**Visual Quiz Delivery** вирішує це автоматично.

---

## 6. Основна цінність

Модуль дає клієнту:

```text
готову вікторину + візуальний контекст + автоматичну доставку
```

Це вже не просто питання з варіантами.  
Це готовий візуальний навчальний пост.

---

## 7. Межі продукту

### 7.1. Що модуль робить

Модуль робить:

- бере вікторину з Quiz Bank;
- аналізує питання, відповіді, тему і рівень;
- створює опис для зображення;
- перевіряє кеш;
- генерує зображення, якщо готового ще немає;
- перевіряє результат;
- зберігає зображення;
- публікує вікторину з картинкою;
- веде журнал доставок і витрат.

### 7.2. Що модуль не робить

Модуль не повинен:

- змінювати сам зміст вікторини без окремого правила;
- вигадувати нові відповіді;
- публікувати неперевірене зображення;
- давати безлімітну генерацію без контролю витрат;
- змішувати брендовані стилі різних клієнтів;
- обходити API/selection engine, status gates, entitlements або quota rules;
- бути окремим хаотичним сервісом поза Quiz Bank.

---

## 8. Режими доставки

```text
delivery_mode:
- text_only
- image_standard
- image_branded
```

### 8.1. `text_only`

Звичайна вікторина без зображення.

### 8.2. `image_standard`

Вікторина з автоматично створеним або кешованим зображенням у стандартному стилі.

### 8.3. `image_branded`

Вікторина з візуалом у стилі клієнта.

Може включати:

- кольори бренду;
- фіксовану композицію;
- стабільний стиль;
- власну візуальну подачу;
- окремий стиль для каналу або бота.

---

## 9. Основні сценарії

### 9.1. Сценарій 1 — текстова доставка

```text
1. Клієнт має тариф text_only
2. Система вибирає вікторину
3. Бот публікує питання і варіанти через Telegram quiz poll
4. Зображення не створюється
```

### 9.2. Сценарій 2 — стандартна візуальна доставка

```text
1. Клієнт має тариф image_standard
2. Система вибирає вікторину
3. Система перевіряє кеш
4. Якщо зображення є — використовує його
5. Якщо зображення немає — генерує нове
6. QA перевіряє результат
7. Бот публікує зображення + quiz poll
```

### 9.3. Сценарій 3 — брендована доставка

```text
1. Клієнт має тариф image_branded
2. Система бере стиль клієнта
3. Створює або знаходить зображення саме під цей стиль
4. Перевіряє якість
5. Публікує вікторину в брендованому форматі
```

### 9.4. Сценарій 4 — збій генерації

```text
1. Генерація не вдалася
2. QA відхилив зображення
3. Ліміт генерацій вичерпано
4. Система не зупиняє доставку
5. Вікторина публікується у text_only або з кешованим fallback-зображенням
```

---

## 10. Ключові правила якості

Модуль повинен дотримуватись таких правил:

1. Зображення має відповідати питанню.
2. Зображення не повинно ламати навчальну логіку.
3. Зображення не повинно випадково підказувати правильну відповідь, якщо це шкодить вікторині.
4. Текст усередині зображення краще не використовувати без потреби.
5. Питання і відповіді мають залишатися у Telegram quiz poll.
6. Погане зображення краще не публікувати.
7. При сумніві система має перейти на безпечний fallback.

## 13. Принцип контролю витрат

Модуль не повинен працювати за принципом безконтрольної генерації.

Потрібно рахувати:

- кількість генерацій;
- кількість кеш-хітів;
- вартість генерації;
- кількість fallback-випадків;
- вартість на одного клієнта;
- маржинальність тарифу.

---

## 14. Кеш як основа прибутковості

Кеш — обов’язкова частина продукту.

Якщо зображення вже створене для:

```text
quiz_item_id + delivery_mode + visual_style + branding_preset + image_version + language
```

система не генерує його повторно.

Це дозволяє:

- зменшити витрати;
- пришвидшити доставку;
- повторно використовувати готовий актив у межах дозволених прав і consumer boundaries;
- масштабувати продукт дешевше.

Брендовані assets повинні бути client-scoped і не можуть повторно використовуватися для іншого клієнта без явного дозволу та окремого branding preset.

## 16. Мінімальні API-можливості

Модуль повинен мати внутрішні можливості:

```text
POST /v1/visual-assets/generate
GET  /v1/visual-assets/{quiz_item_id}
POST /v1/deliveries/visual
GET  /v1/consumers/{consumer_id}/visual-settings
PATCH /v1/consumers/{consumer_id}/visual-settings
```

Це не означає, що всі ці маршрути мають бути одразу публічними.  
Частина може бути лише внутрішньою для backend-процесу.

Якщо capability стає public API, вона повинна залишатися versioned і не обходити auth, entitlement, quota, status filtering або delivery logging.

## 17. Ролі системи

### Quiz Bank

Платформне ядро і governed source of truth для delivery-eligible вікторини через API/selection engine, а не прямий raw CSV access.

### Visual Prompt Builder

Створює опис для зображення.

### Image Generator

Генерує зображення.

### Visual QA

Перевіряє зміст і якість.

### Asset Storage

Зберігає зображення.

### Delivery Worker

Публікує у Telegram.

### Billing / Entitlement

Перевіряє тариф, ліміти і права клієнта.

## 18. Метрики успіху

Модуль вважається успішним, якщо:

- клієнти обирають дорожчий visual-тариф;
- візуальні вікторини мають вищу залученість;
- кеш зменшує витрати;
- fallback не ламає доставку;
- QA блокує погані зображення;
- генерація не створює фінансового ризику;
- система стабільно працює в автоматичному режимі.

## 20. Рішення по продукту

Прийняте рішення:

**Visual Quiz Delivery будується як premium-layer над API Quiz Bank.**

Він не є окремим випадковим сервісом.

Продуктова структура:

```text
API Quiz Bank
│
├── Text Quiz Delivery
│
├── Visual Quiz Delivery
│   ├── Standard Visual
│   └── Branded Visual
│
└── Billing / Entitlements / Quotas
```

---

## 21. Непорушні принципи

1. Платформне ядро Quiz Bank залишається джерелом правди для delivery-eligible вікторин.
2. Зображення не повинно змінювати суть вікторини.
3. Візуальна доставка є платною цінністю.
4. Генерація завжди має бути під контролем витрат.
5. Кеш є обов’язковим.
6. Погане зображення не публікується.
7. Доставка не повинна падати через проблему з картинкою.
8. Клієнт має чітко обирати режим доставки.
9. Брендовані стилі клієнтів не змішуються.
10. Модуль має бути готовий до SaaS-масштабування.

---

## 22. Фінальний висновок

**Visual Quiz Delivery** — це сильний комерційний модуль для API Quiz Bank.

Він дозволяє продавати не просто банк питань, а готовий автоматизований формат навчального контенту:

```text
вікторина + візуальний контекст + доставка + контроль якості
```

Це підвищує цінність продукту, відкриває новий тариф і робить Quiz Bank ближчим до повноцінного SaaS-рішення для власників каналів, ботів і освітніх проєктів.
